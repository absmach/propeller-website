---
title: Embedded Proplet
description: WebAssembly runtime for microcontroller-based edge devices using Zephyr RTOS and WAMR
---

The Embedded Proplet brings WebAssembly execution to microcontrollers like the ESP32-S3. It uses the WebAssembly Micro Runtime (WAMR) on Zephyr RTOS, enabling task execution on resource-constrained devices with as little as 128 KB RAM.

## Why the Embedded Proplet

### Extending to Constrained Devices

The standard [Proplet](/docs/proplet) runs on Linux with Wasmtime, requiring ~100 MB RAM minimum. The Embedded Proplet targets microcontrollers with kilobytes of memory, enabling Propeller to reach sensors, actuators, and other embedded devices at the true edge.

### Native RTOS Integration

Built on Zephyr RTOS, the Embedded Proplet integrates with hardware-level features like WiFi drivers, interrupt handling, and real-time scheduling. This is essential for devices that interact directly with physical hardware.

### Same Protocol, Different Runtime

The Embedded Proplet uses the same MQTT messaging protocol and topic structure as the standard Proplet. The Manager sees it as just another Proplet—it publishes discovery messages, receives tasks, and reports results identically.

## Prerequisites

Before building the Embedded Proplet, set up the development environment as described in the [Zephyr Guide](/docs/zephyr). You'll need:

- Zephyr SDK (v0.16.0+)
- ESP-IDF (v5.1+) for ESP32-S3
- West build system
- WAMR submodule initialized

```bash
cd propeller/embed-proplet
git submodule update --init --recursive
```

## Building and Flashing

### Configure Device Identity

Edit `src/main.c` with your device settings:

```c
#define WIFI_SSID "your_wifi_ssid"
#define WIFI_PSK "your_wifi_password"
#define PROPLET_ID "proplet-esp32-001"
#define DOMAIN_ID "your_domain_id"
#define CHANNEL_ID "your_channel_id"
```

### Build for ESP32-S3

```bash
cd embed-proplet
west build -b esp32s3_devkitc/esp32s3/procpu -p auto .
```

### Flash the Device

```bash
west flash
```

### Monitor Output

```bash
west espressif monitor
```

## Architecture

The Embedded Proplet consists of several C modules that handle different responsibilities.

### Source Components

| File | Purpose |
|------|---------|
| `main.c` | Entry point, WiFi/MQTT initialization, heartbeat loop |
| `wifi_manager.c` | WiFi connection management |
| `mqtt_client.c` | MQTT protocol, discovery, subscriptions |
| `wasm_handler.c` | WAMR module loading and execution |
| `task_monitor.c` | Task state tracking and metrics |

### WAMR Integration

The CMake configuration sets up WAMR for the Zephyr/Xtensa target:

```cmake
set(WAMR_BUILD_PLATFORM "zephyr")
set(WAMR_BUILD_TARGET "XTENSA")
set(WAMR_BUILD_INTERP 1)
set(WAMR_BUILD_AOT 1)
set(WAMR_BUILD_GLOBAL_HEAP_SIZE 40960)  # 40 KB
```

WAMR supports both interpreter and AOT modes. AOT-compiled modules are more compact and faster but require pre-compilation for the target architecture.

### Memory Budget

The ESP32-S3 has 520 KB SRAM. Typical allocation:

| Subsystem | Memory |
|-----------|--------|
| Zephyr kernel | ~100 KB |
| WiFi/MQTT stack | ~150 KB |
| Logging buffers | ~20 KB |
| WAMR + modules | ~40 KB |
| User heap | ~210 KB |

Memory limits from `CMakeLists.txt`:
- Global WAMR heap: 40 KB
- Per-module stack: 16 KB
- Per-module heap: 16 KB

## Configuration

### Zephyr Configuration (`prj.conf`)

Key settings in the project configuration:

```ini
# Networking
CONFIG_NETWORKING=y
CONFIG_NET_TCP=y
CONFIG_NET_SOCKETS=y

# WiFi
CONFIG_WIFI=y
CONFIG_WIFI_ESP32=y

# MQTT
CONFIG_MQTT_LIB=y
CONFIG_MQTT_LIB_TLS=n

# Memory
CONFIG_HEAP_MEM_POOL_SIZE=65536
CONFIG_MAIN_STACK_SIZE=8192

# Logging
CONFIG_LOG=y
CONFIG_LOG_DEFAULT_LEVEL=3
```

### Board-Specific Configuration

Create `boards/esp32s3_devkitc.conf` for board-specific overrides:

```ini
CONFIG_ESP32_WIFI_STA_RECONNECT=y
CONFIG_ESP32_WIFI_STA_AUTO_DHCPV4=y
```

### Runtime Constants

In `src/main.c`:

```c
#define PROPLET_LIVELINESS_INTERVAL_MS 10000   // Heartbeat frequency
#define PROPLET_METRICS_INTERVAL_MS 30000      // System metrics frequency
#define PROPLET_TASK_METRICS_INTERVAL_MS 10000 // Task metrics frequency
```

## MQTT Communication

The Embedded Proplet uses the same MQTT topic structure as the standard Proplet. See the [Proplet MQTT Topics](/docs/proplet#mqtt-topics) section for the complete topic reference.

### Discovery

On startup, the Embedded Proplet publishes a discovery message to register with the Manager:

```c
publish_discovery(domain_id, PROPLET_ID, channel_id);
```

The Manager receives this and adds the Proplet to its fleet.

### Heartbeats

The main loop publishes periodic messages:

- **Alive messages** (every 10s): Indicate the Proplet is operational
- **System metrics** (every 30s): Memory usage, uptime
- **Task metrics** (every 10s): Active task status

### WASM Reception

The Embedded Proplet receives WASM modules either:
- **Directly via MQTT** for small modules
- **Via chunked transfer** from the [Proxy](/docs/proxy) for larger modules

The `wasm_handler` reassembles chunks and loads the module into WAMR.

## Testing Execution

Create a task for the Embedded Proplet:

```bash
curl -X POST "http://localhost:7070/tasks" \
-H "Content-Type: application/json" \
-d '{"name": "add", "inputs": [10, 20]}'
```

Upload a WASM module built for WAMR:

```bash
curl -X PUT "http://localhost:7070/tasks/<task_id>/upload" \
-F 'file=@build/addition.wasm'
```

Start the task:

```bash
curl -X POST "http://localhost:7070/tasks/<task_id>/start"
```

Monitor execution on the device:


## Troubleshooting

### CMake Threads Support Error

**Error:** `CMakeLists.txt:XX Could not find Threads`

**Solution:** Add to `prj.conf`:
```ini
CONFIG_PTHREAD_IPC=y
CONFIG_POSIX_API=y
```

### MQTT Connection Failures

**Symptom:** `MQTT connect failed` in logs

**Debug steps:**
1. Verify WiFi connected: look for `WiFi connected to SSID` log
2. Check configuration matches SuperMQ broker settings
3. Enable debug logging:
   ```ini
   CONFIG_MQTT_LOG_LEVEL_DBG=y
   CONFIG_NET_LOG=y
   ```

### Out of Memory

**Symptom:** WASM module fails to load with memory errors

**Solutions:**
1. Use AOT compilation for smaller module size
2. Increase `WAMR_BUILD_GLOBAL_HEAP_SIZE` in `CMakeLists.txt`
3. Reduce heap for other subsystems in `prj.conf`

### Configuration Not Taking Effect

**Symptom:** Settings in `prj.conf` are ignored

**Solution:** Clean rebuild:
```bash
rm -rf build
west build -b esp32s3_devkitc/esp32s3/procpu -p auto .
```

Check for conflicts with board-specific `*.conf` files.

## Porting to Other Hardware

WAMR supports multiple architectures. To port the Embedded Proplet:

### 1. Update CMakeLists.txt

```cmake
# For ARM Cortex-M (nRF52840, STM32F4):
set(BOARD nrf52840dk_nrf52840)
set(WAMR_BUILD_TARGET "THUMB")

# For RISC-V (ESP32-C3):
set(BOARD esp32c3_devkitm)
set(WAMR_BUILD_TARGET "RISCV32")
```

### 2. Create Board Configuration

Add `boards/<board_name>.conf` with board-specific settings.

### 3. Adjust Memory Settings

Different boards have different RAM. Scale `HEAP_MEM_POOL_SIZE` and `WAMR_BUILD_GLOBAL_HEAP_SIZE` accordingly.

### 4. Build and Flash

```bash
rm -rf build
west build -b <board_name> -p auto .
west flash
```

### Supported Architectures

| Architecture | WAMR Target | Example Boards |
|-------------|-------------|----------------|
| Xtensa | `XTENSA` | ESP32, ESP32-S3 |
| ARM Cortex-M | `THUMB` | nRF52840, STM32F4 |
| RISC-V 32 | `RISCV32` | ESP32-C3 |
| RISC-V 64 | `RISCV64` | SiFive |

### Hardware Requirements

| Requirement | Minimum |
|------------|---------|
| RAM | 128 KB |
| Flash | 512 KB |
| Networking | WiFi, Ethernet, or BLE |
| Zephyr support | Board in Zephyr's board list |
