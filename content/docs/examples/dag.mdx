This example demonstrates DAG (Directed Acyclic Graph) workflow execution in Propeller. DAG workflows allow you to define task dependencies where child tasks only execute after their parent tasks complete successfully.

**What you'll learn:**
- Creating workflows with task dependencies
- Specifying task IDs for dependency references
- Automatic scheduling of dependent tasks
- Conditional execution with `run_if`

## Prerequisites

Ensure you have:
- Propeller services running (`make start-supermq`)
- A [provisioned config](/docs/getting-started#step-3-provision-propeller) with valid credentials
- The `addition.wasm` module built

## Create Workflow

Create a DAG workflow with three tasks. Task 2 and Task 3 both depend on Task 1, so they'll execute in parallel after Task 1 completes.

**Important:** Specify task IDs explicitly so dependencies can reference them:

```bash
curl -X POST "http://localhost:7070/workflows" \
-H "Content-Type: application/json" \
-d '{
  "tasks": [
    {
      "id": "task-001",
      "name": "main",
      "inputs": [10, 20]
    },
    {
      "id": "task-002",
      "name": "main",
      "inputs": [5, 15],
      "depends_on": ["task-001"]
    },
    {
      "id": "task-003",
      "name": "main",
      "inputs": [100, 200],
      "depends_on": ["task-001"]
    }
  ]
}'
```

## Upload WASM

Upload the WASM module to each task:

```bash
curl -X PUT "http://localhost:7070/tasks/task-001/upload" \
-F "file=@build/addition.wasm"

curl -X PUT "http://localhost:7070/tasks/task-002/upload" \
-F "file=@build/addition.wasm"

curl -X PUT "http://localhost:7070/tasks/task-003/upload" \
-F "file=@build/addition.wasm"
```

Each upload returns the updated task with the embedded WASM (base64 encoded in the `file` field).

## Conditional Execution

Tasks can run conditionally based on dependency outcomes using the `run_if` field:

```bash
curl -X POST "http://localhost:7070/workflows" \
-H "Content-Type: application/json" \
-d '{
  "tasks": [
    {
      "id": "pipeline-001",
      "name": "main",
      "inputs": [10, 20]
    },
    {
      "id": "pipeline-002",
      "name": "main",
      "inputs": [5, 15],
      "depends_on": ["pipeline-001"],
      "run_if": "success"
    },
    {
      "id": "pipeline-003",
      "name": "main",
      "inputs": [0, 0],
      "depends_on": ["pipeline-001"],
      "run_if": "failure"
    }
  ]
}'
```

| `run_if` value | Behavior |
|----------------|----------|
| `success` | Task runs only if all dependencies completed successfully (state 3) |
| `failure` | Task runs only if any dependency failed (state 4) |
| (omitted) | Task runs when all dependencies complete, regardless of outcome |

In the example above:
- **pipeline-002** runs only if pipeline-001 succeeds (happy path)
- **pipeline-003** runs only if pipeline-001 fails (error handling)

## Task States

| State | Value | Description |
|-------|-------|-------------|
| Pending | 0 | Task created but not scheduled |
| Scheduled | 1 | Task queued for execution |
| Running | 2 | Task currently executing |
| Completed | 3 | Task finished successfully |
| Failed | 4 | Task execution failed |
| Skipped | 5 | Task skipped due to `run_if` condition |
| Interrupted | 6 | Task was cancelled |

## DAG Validation

The manager validates workflows before accepting them:

1. **Dependency existence** - All task IDs in `depends_on` must exist in the workflow
2. **Cycle detection** - No circular dependencies allowed (A→B→C→A is invalid)
3. **`run_if` values** - Must be either `success` or `failure`

Invalid workflows return an error:

```json
{
  "error": "dependency validation failed: missing dependency: task X depends on Y which does not exist"
}
```
