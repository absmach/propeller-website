This example demonstrates [DAG (Directed Acyclic Graph) workflow](/docs/task-scheduling#workflows) execution in Propeller. DAG workflows allow you to define task dependencies where child tasks only execute after their parent tasks complete successfully.

## Prerequisites

Ensure you have:
- Propeller services running (`make start-supermq`)
- A [provisioned config](/docs/getting-started#step-3-provision-propeller) with valid credentials
- The `addition.wasm` module built

## Source Code

This example uses the [addition-wat](https://github.com/absmach/propeller/tree/main/examples/addition-wat) example, which exports a `main` function that adds two i32 integers:

```lisp
(module
    (func (export "main") (param i32 i32) (result i32)
        local.get 0
        local.get 1
        i32.add))
```

## Build WASM

Propeller executes tasks as WebAssembly modules, so we need to compile the source code to `.wasm` format before uploading.

Build the addition module from WAT (WebAssembly Text format):

```bash
cd propeller
wat2wasm examples/addition-wat/addition.wat -o build/addition.wasm
```

The command produces no output on success.

Verify the build:

```bash
file build/addition.wasm
```

Your output should look like this:

```text
build/addition.wasm: WebAssembly (wasm) binary module version 0x1 (MVP)
```

This confirms the file is a valid WebAssembly binary. "Version 0x1 (MVP)" indicates WebAssembly 1.0, the first standardized version supported by all runtimes.

## Create Workflow

Create a DAG workflow with three tasks. Three tasks is the minimum needed to demonstrate [parallel fan-out](/docs/task-scheduling#fan-out-and-fan-in): Task 1 is the root, and Task 2 and Task 3 both depend on it. When Task 1 completes, Tasks 2 and 3 start simultaneously.

![Example DAG](/images/DAG/Example-DAG.svg)

Task IDs are auto-generated by Propeller if omitted. In this example, we explicitly set `"id": "task-001"` so that `task-002` and `task-003` can reference it in their `depends_on` arrays:

```bash
curl -X POST "http://localhost:7070/workflows" \
-H "Content-Type: application/json" \
-d '{
  "tasks": [
    {
      "id": "task-001",
      "name": "main",
      "inputs": [10, 20]
    },
    {
      "id": "task-002",
      "name": "main",
      "inputs": [5, 15],
      "depends_on": ["task-001"]
    },
    {
      "id": "task-003",
      "name": "main",
      "inputs": [100, 200],
      "depends_on": ["task-001"]
    }
  ]
}'
```

## Upload WASM

Since we built `addition.wasm` locally, we upload it directly to each task. For production workflows, you can use `image_url` instead—see [WASM Module Deployment](/docs/task-scheduling#wasm-module-deployment).

### Task 1

```bash
curl -X PUT "http://localhost:7070/tasks/task-001/upload" \
-F "file=@build/addition.wasm"
```

Your output should look like this:

```json
{
  "id": "task-001",
  "name": "main",
  "state": 0,
  "file": "AGFzbQEAAAABBwFgAn9/AX8DAgEABwgBBG1haW4AAAoJAQcAIAAgAWoL",
  "cli_args": null,
  "inputs": [10, 20],
  "daemon": false,
  "encrypted": false,
  "workflow_id": "33b97dd3-f719-40f0-8a45-87254c27406f",
  "start_time": "0001-01-01T00:00:00Z",
  "finish_time": "0001-01-01T00:00:00Z",
  "created_at": "2026-03-01T15:21:52.857522851Z",
  "updated_at": "2026-03-01T15:21:55.123456789Z",
  "next_run": "0001-01-01T00:00:00Z"
}
```

The `file` field now contains the WASM binary encoded as base64. This is the actual module that will be sent to the proplet for execution. The `updated_at` timestamp reflects when the upload occurred.

### Tasks 2 and 3

Repeat the same upload for the remaining tasks. The response structure is identical—only the task ID, inputs, and `depends_on` fields differ:

```bash
curl -X PUT "http://localhost:7070/tasks/task-002/upload" \
-F "file=@build/addition.wasm"

curl -X PUT "http://localhost:7070/tasks/task-003/upload" \
-F "file=@build/addition.wasm"
```

## Start Workflow

Start the root task (`task-001`). The manager automatically schedules dependent tasks (`task-002` and `task-003`) when their dependencies complete:

```bash
curl -X POST "http://localhost:7070/tasks/task-001/start"
```

Your output should look like this:

```json
{ "started": true }
```

## Conditional Execution

Tasks can run conditionally based on dependency outcomes using the `run_if` field:

```bash
curl -X POST "http://localhost:7070/workflows" \
-H "Content-Type: application/json" \
-d '{
  "tasks": [
    {
      "id": "pipeline-001",
      "name": "main",
      "inputs": [10, 20]
    },
    {
      "id": "pipeline-002",
      "name": "main",
      "inputs": [5, 15],
      "depends_on": ["pipeline-001"],
      "run_if": "success"
    },
    {
      "id": "pipeline-003",
      "name": "main",
      "inputs": [0, 0],
      "depends_on": ["pipeline-001"],
      "run_if": "failure"
    }
  ]
}'
```

## DAG Validation

The manager validates workflows before accepting them:

1. **Dependency existence** - All task IDs in `depends_on` must exist in the workflow
2. **Cycle detection** - No circular dependencies allowed (A→B→C→A is invalid)
3. **`run_if` values** - Must be either `success` or `failure`

Invalid workflows return an error:

```json
{
  "error": "dependency validation failed: missing dependency: task X depends on Y which does not exist"
}
```
