This example demonstrates two orchestration methods in Propeller:

1. **Workflows** — Define explicit task dependencies using a DAG structure. Tasks specify which other tasks they depend on via `depends_on`.
2. **Jobs** — Group tasks with a shared execution mode (`parallel` or `sequential`). No explicit dependencies—the mode controls execution order.

Both use the same underlying tasks but differ in how execution order is determined.

---

## Part 1: DAG Workflow

This section demonstrates [DAG (Directed Acyclic Graph) workflow](/docs/task-scheduling#workflows) execution. Workflows allow you to define task dependencies where child tasks only execute after their parent tasks complete successfully.

### Prerequisites

Ensure you have:
- Propeller services running (`make start-supermq`)
- A [provisioned config](/docs/getting-started#step-3-provision-propeller) with valid credentials
- The `addition.wasm` module built

### Source Code

This example uses the [addition-wat](https://github.com/absmach/propeller/tree/main/examples/addition-wat) example, which exports a `main` function that adds two i32 integers:

```lisp
(module
    (func (export "main") (param i32 i32) (result i32)
        local.get 0
        local.get 1
        i32.add))
```

### Build WASM

Propeller executes tasks as WebAssembly modules, so we need to compile the source code to `.wasm` format before uploading.

Build the addition module from WAT (WebAssembly Text format):

```bash
cd propeller
wat2wasm examples/addition-wat/addition.wat -o build/addition.wasm
```

The command produces no output on success.

Verify the build:

```bash
file build/addition.wasm
```

Your output should look like this:

```text
build/addition.wasm: WebAssembly (wasm) binary module version 0x1 (MVP)
```

This confirms the file is a valid WebAssembly binary. "Version 0x1 (MVP)" indicates WebAssembly 1.0, the first standardized version supported by all runtimes.

### Create Workflow

Create a DAG workflow with three tasks. Three tasks is the minimum needed to demonstrate [parallel fan-out](/docs/task-scheduling#fan-out-and-fan-in): Task 1 is the root, and Task 2 and Task 3 both depend on it. When Task 1 completes, Tasks 2 and 3 start simultaneously.

![Example DAG](../images/DAG/Example-DAG.svg)

Task IDs are auto-generated by Propeller if omitted. In this example, we explicitly set `"id": "task-001"` so that `task-002` and `task-003` can reference it in their `depends_on` arrays:

```bash
curl -X POST "http://localhost:7070/workflows" \
-H "Content-Type: application/json" \
-d '{
  "tasks": [
    {
      "id": "task-001",
      "name": "main",
      "inputs": [10, 20]
    },
    {
      "id": "task-002",
      "name": "main",
      "inputs": [5, 15],
      "depends_on": ["task-001"]
    },
    {
      "id": "task-003",
      "name": "main",
      "inputs": [100, 200],
      "depends_on": ["task-001"]
    }
  ]
}'
```

```bash

```

Your output should look like this:

```json

```

## Part 2: Job

This section demonstrates [job-based execution](/docs/task-scheduling#jobs). Jobs group tasks with a shared execution mode—no explicit dependencies. The mode controls whether tasks run in parallel or sequentially.

### Create Job

```bash

-H "Content-Type: application/json" \

  "execution_mode": "parallel",

    {
      "id": "job-task-001",
```

### Execution Mode Comparison

| Mode | Behavior | Start Times (this example) |
|------|----------|---------------------------|
| `parallel` | All tasks start simultaneously | `.452`, `.468`, `.485` (33ms spread) |
| `sequential` | Tasks start one after another | `.099`, `.120`, `.153` (staggered) |

With fast-executing tasks, the timing difference is subtle. For long-running tasks, sequential mode ensures each task completes before the next starts (fail-fast behavior).

For a comparison of workflows vs jobs and detailed job configuration options, see [Workflow vs Job Comparison](/docs/task-scheduling#workflow-vs-job-comparison).

## Troubleshooting

Common issues when working with DAG workflows and jobs.

### Task Stuck in Pending

If a task remains in `state: 0` (Pending) and doesn't start:

**1. Check proplet availability**

At least one proplet must be alive to execute tasks:

```bash
curl http://localhost:7070/proplets | jq '.proplets[] | {id, alive}'
```

**2. Verify dependencies are satisfied**

Tasks with unmet dependencies stay pending:

```bash
curl http://localhost:7070/tasks/task-abc | jq '{depends_on, state}'
```

**3. Check priority isn't too low**

Higher-priority tasks get dispatched first:

```bash
curl http://localhost:7070/tasks | jq '.tasks | sort_by(.priority) | reverse'
```

### Workflow Tasks Not Starting

**1. Verify DAG validation passed**

Invalid DAGs (circular dependencies, missing task references) are rejected:

```bash
curl http://localhost:7070/workflows/workflow-abc | jq '.validation_errors'
```

**2. Check run_if conditions**

Tasks may be skipped based on dependency outcomes:
- `run_if: success` but a dependency failed → task will be `Skipped`
- `run_if: failure` but dependencies succeeded → task will be `Skipped`

**3. Verify workflow was started**

Workflows don't auto-start—you must start the root task:

```bash
curl -X POST http://localhost:7070/tasks/{root-task-id}/start
```

### Cron Tasks Not Running

**1. Verify cron expression is valid**

Use a cron validator or parser:

```bash
# Example: check if expression would run
echo "0 * * * *"  # Every hour at minute 0
```

**2. Check is_recurring is true**

Tasks without `is_recurring: true` only run once:

```bash
curl http://localhost:7070/tasks/task-abc | jq '{schedule, is_recurring, next_run}'
```

**3. Verify timezone is correct**

Cron evaluation uses the specified timezone:

```bash
TZ="America/New_York" date
```
