This is a federated learning example. Proplets train models locally on their private data, and the coordinator aggregates updates using the FedAvg algorithm. The model never leaves the device — only weight updates are shared.

For a comprehensive guide on federated learning architecture, motivation, and the complete training lifecycle, see the [Federated Machine Learning](/docs/federated-learning) documentation.

## Prerequisites

- Docker and Docker Compose
- Python 3 (for provisioning script)
- `docker/.env` file with SuperMQ configuration
- GitHub account with Personal Access Token (PAT) for GHCR (if using GHCR for WASM storage)

## Build and Start Services

The FL demo extends the base SuperMQ + Propeller stack with FL-specific services. You must use both compose files together.

:::warning Important
The FL demo compose file (`examples/fl-demo/compose.yaml`) is an **extension** to the base compose file (`docker/compose.yaml`). Running it standalone will fail with network errors like `service "local-data-store" refers to undefined network supermq-base-net`.
:::

### Build Images

Manager and proplet must be built from source as they include FL endpoints:

```bash
cd propeller
docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env build manager proplet proplet-2 proplet-3 coordinator-http
```

### Start All Services

```bash
docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env up -d
```

This starts:
- Full SuperMQ production stack (Auth, Domains, Clients, Channels, MQTT Adapter, Nginx)
- FL-specific services (Model Registry, Aggregator, Local Data Store, Coordinator)
- Propeller services (Manager, Proplets, Proxy)

Wait a few seconds for services to start, then verify:

```bash
docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env ps
```

Your output should show all services running.

## Provision SuperMQ Resources

Before the manager and proplets can connect to MQTT, you must provision the necessary SuperMQ resources (domain, channel, and clients).

:::note One-Time Setup
Provisioning creates persistent data in SuperMQ databases. Only needs to be repeated if you remove Docker volumes (e.g., `docker compose down -v`).
:::

Install Python dependencies:

```bash
pip install -r examples/fl-demo/requirements.txt
```

Run the provisioning script:

(cd examples/fl-demo && python3 provision-smq.py)

## Recreate Services After Provisioning

After provisioning, you must **recreate** (not just restart) the services to pick up the new credentials:

```bash
docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env up -d --force-recreate manager coordinator-http proplet proplet-2 proplet-3 proxy local-data-store
```

Verify services are running:

```bash
docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env ps manager coordinator-http proplet proplet-2 proplet-3 proxy
```

Check health endpoints:

```bash
curl http://localhost:7070/health
curl http://localhost:8086/health
```

Your output should look like this:

```
OK
OK
```

## Check Initial Model

Verify the model registry has the initial model (version 0):

```bash
curl -s http://localhost:8084/models/0 | jq .
```

Your output should look like this:

```json
{
  "b": 0,
  "version": 0,
  "w": [0, 0, 0]
}
```

If no model exists, create the initial model:

```bash
curl -X POST http://localhost:8084/models \
  -H "Content-Type: application/json" \
  -d '{
    "version": 0,
    "model": {
      "w": [0.0, 0.0, 0.0],
      "b": 0.0
    }
  }'
```

## Verify Training Data

The Local Data Store auto-seeds datasets for configured proplet UUIDs on startup. Verify datasets exist:

```bash
export PROPLET_CLIENT_ID=$(grep '^PROPLET_CLIENT_ID=' docker/.env | grep -v '=""' | tail -1 | cut -d '=' -f2 | tr -d '"')
curl -s http://localhost:8083/datasets/$PROPLET_CLIENT_ID | jq '.schema, .proplet_id, .size'
```

Your output should look like this:

```json
"fl-demo-dataset-v1"
"<your-proplet-uuid>"
64
```

## Build Wasm

Build the FL client to WASM:

```bash
cd examples/fl-demo/client-wasm
GOTOOLCHAIN=go1.25.5 GOOS=wasip2 GOARCH=wasm go build -o fl-client.wasm fl-client.go
cd ../../..
```

## Push WASM to Registry

Push the WASM to GitHub Container Registry (GHCR):

```bash
# Login to GHCR
echo $GITHUB_TOKEN | docker login ghcr.io -u YOUR_GITHUB_USERNAME --password-stdin

# Push using ORAS
docker run --rm \
  -v "$(pwd)/examples/fl-demo/client-wasm:/workspace" \
  -w /workspace \
  -v "$HOME/.docker/config.json:/root/.docker/config.json:ro" \
  ghcr.io/oras-project/oras:v1.3.0 \
  push ghcr.io/YOUR_GITHUB_USERNAME/fl-client-wasm:latest \
  fl-client.wasm:application/wasm
```

Your output should look like this:

```
Pushed [registry] ghcr.io/YOUR_GITHUB_USERNAME/fl-client-wasm:latest
ArtifactType: application/vnd.unknown.artifact.v1
Digest: sha256:...
```

Alternatively, push to local registry:

```bash
docker run --rm \
  -v $(pwd)/examples/fl-demo/client-wasm:/workspace \
  -w /workspace \
  --network host \
  ghcr.io/oras-project/oras:latest \
  push localhost:5000/fl-client-wasm:latest \
  fl-client.wasm:application/wasm
```

## Configure GHCR Authentication

Add GHCR credentials to `docker/.env`:

```bash
PROXY_AUTHENTICATE=true
PROXY_REGISTRY_URL=ghcr.io
PROXY_REGISTRY_USERNAME=YOUR_GITHUB_USERNAME
PROXY_REGISTRY_PASSWORD=ghp_xxxxx  # Your GitHub PAT with read:packages scope
```

Recreate proxy to pick up new credentials:

```bash
docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env up -d --force-recreate proxy
```

## Trigger a Federated Learning Round

Export the proplet client IDs:

```bash
export PROPLET_CLIENT_ID=$(grep '^PROPLET_CLIENT_ID=' docker/.env | grep -v '=""' | tail -1 | cut -d '=' -f2 | tr -d '"')
export PROPLET_2_CLIENT_ID=$(grep '^PROPLET_2_CLIENT_ID=' docker/.env | cut -d '=' -f2 | tr -d '"')
export PROPLET_3_CLIENT_ID=$(grep '^PROPLET_3_CLIENT_ID=' docker/.env | cut -d '=' -f2 | tr -d '"')

# Verify they're set (should show UUIDs)
echo "PROPLET_CLIENT_ID=$PROPLET_CLIENT_ID"
echo "PROPLET_2_CLIENT_ID=$PROPLET_2_CLIENT_ID"
echo "PROPLET_3_CLIENT_ID=$PROPLET_3_CLIENT_ID"
```

Configure and start the FL experiment:

```bash
curl -X POST http://localhost:7070/fl/experiments \
  -H "Content-Type: application/json" \
  -d "{
    \"experiment_id\": \"exp-r-$(date +%s)\",
    \"round_id\": \"r-$(date +%s)\",
    \"model_ref\": \"fl/models/global_model_v0\",
    \"participants\": [\"$PROPLET_CLIENT_ID\", \"$PROPLET_2_CLIENT_ID\", \"$PROPLET_3_CLIENT_ID\"],
    \"hyperparams\": {\"epochs\": 1, \"lr\": 0.01, \"batch_size\": 16},
    \"k_of_n\": 3,
    \"timeout_s\": 60,
    \"task_wasm_image\": \"ghcr.io/YOUR_GITHUB_USERNAME/fl-client-wasm:latest\"
  }"
```

Your output should look like this:

```json
{
  "experiment_id": "exp-r-1709309984",
  "round_id": "r-1709309984",
  "status": "configured"
}
```

After aggregation, verify the model was updated:

```bash
curl -s http://localhost:8084/models | jq .
```

Your output should look like this:

```json
{
  "versions": [0, 1]
}
```

Fetch the aggregated model:

```bash
curl -s http://localhost:8084/models/1 | jq .
```

Your output should look like this:

```json
{
  "b": 4.95,
  "version": 1,
  "w": [2.1, 2.9, 1.05]
}
```

The weights (`w`) and bias (`b`) should be non-zero, indicating successful training and aggregation.

## Run Multiple Rounds

To run additional rounds, trigger another experiment with the new model version:

```bash
curl -X POST http://localhost:7070/fl/experiments \
  -H "Content-Type: application/json" \
  -d "{
    \"experiment_id\": \"exp-r-$(date +%s)\",
    \"round_id\": \"r-$(date +%s)\",
    \"model_ref\": \"fl/models/global_model_v1\",
    \"participants\": [\"$PROPLET_CLIENT_ID\", \"$PROPLET_2_CLIENT_ID\", \"$PROPLET_3_CLIENT_ID\"],
    \"hyperparams\": {\"epochs\": 1, \"lr\": 0.01, \"batch_size\": 16},
    \"k_of_n\": 3,
    \"timeout_s\": 60,
    \"task_wasm_image\": \"ghcr.io/YOUR_GITHUB_USERNAME/fl-client-wasm:latest\"
  }"
```

Each round produces a new model version that builds on the previous one.

## Troubleshooting

### Services Fail to Start

If you see `undefined network supermq-base-net`:
- You're running the FL compose file standalone. Use both compose files:
  ```bash
  docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env up -d
  ```

### Environment Variables Not Set

If you see warnings about `PROPLET_CLIENT_ID` not set:
- Run the provisioning script first
- Then recreate services with `--force-recreate`

### Manager Returns 404

If `/fl/experiments` returns 404:
- Manager wasn't built from source. Rebuild:
  ```bash
  docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env build manager
  docker compose -f docker/compose.yaml -f examples/fl-demo/compose.yaml --env-file docker/.env up -d manager
  ```

### Weights Remain Zero

If `w` remains `[0,0,0]` after training:
- Check proplet logs for dataset loading errors
- Verify datasets exist in Local Data Store
- Rebuild WASM client after code changes
