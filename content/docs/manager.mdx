## Overview

The Manager is the central orchestrator written in Go. It is responsible for the complete lifecycle of WebAssembly workloads: creating tasks, scheduling them to proplets, tracking their execution states, and storing results. It also manages jobs, workflows, cron-scheduled tasks, and federated learning experiments.

The Manager exposes a REST HTTP API and communicates with proplets over MQTT using the SuperMQ message broker.

The Manager exposes its REST API on port `7070` by default.

| ------ | ---- | ----------- |

| `GET` | `/proplets/{id}` | Get a specific proplet by ID |
| `DELETE` | `/proplets/{id}` | Remove a proplet record |
| `GET` | `/proplets/{id}/metrics` | Get stored metrics for a proplet |

### Tasks

| Method | Path | Description |
| ------ | ---- | ----------- |
| `POST` | `/tasks` | Create a new task |
| `GET` | `/tasks` | List all tasks (supports `?offset=` and `?limit=`) |
| `GET` | `/tasks/{id}` | Get a specific task by ID |
| `PUT` | `/tasks/{id}` | Update a task (e.g. set `file` as base64-encoded Wasm) |
| `PUT` | `/tasks/{id}/upload` | Upload a `.wasm` file directly (`multipart/form-data`, max 100 MB) |
| `DELETE` | `/tasks/{id}` | Delete a task |
| `POST` | `/tasks/{id}/start` | Start a task (schedules it to a proplet) |
| `POST` | `/tasks/{id}/stop` | Stop a running task |
| `GET` | `/tasks/{id}/metrics` | Get stored process metrics for a task |
| `GET` | `/tasks/{id}/results` | Get stored execution results for a task |

#### Task Fields

Tasks support a wide range of fields for different execution scenarios:

| Field | Type | Description |
| ----- | ---- | ----------- |
| `name` | string | Human-readable task name |
| `image_url` | string | OCI registry URL for the Wasm binary |
| `file` | []byte | Base64-encoded Wasm binary (alternative to `image_url`) |
| `cli_args` | []string | Command-line arguments passed to the Wasm module |
| `inputs` | []uint64 | Input values for simple tasks (e.g., compute demos) |
| `env` | map | Environment variables for the Wasm module |
| `daemon` | bool | If true, task runs continuously until stopped |
| `encrypted` | bool | If true, the Wasm binary is encrypted (requires KBS) |
| `kbs_resource_path` | string | KBS resource path for encrypted binaries (TEE only) |
| `monitoring_profile` | object | CPU/memory usage monitoring configuration |
| `priority` | int | Dispatch priority (0–100, default 50) |
| `schedule` | string | Cron expression for scheduled tasks |
| `timezone` | string | Timezone for scheduled tasks (default UTC) |
| `is_recurring` | bool | If true, task re-runs according to `schedule` |

## Jobs

Jobs group multiple tasks together and execute them with a common strategy.

**Create a job:**

```bash
curl -X POST "http://localhost:7070/jobs" \
-H "Content-Type: application/json" \
-d '{
  "name": "batch-process",
  "execution_mode": "sequential",
  "tasks": [
    {"name": "step-1", "image_url": "docker.io/myorg/step1:v1"},
    {"name": "step-2", "image_url": "docker.io/myorg/step2:v1"}
  ]
}'
```

**Execution modes:**

| Mode | Behavior |
| ---- | -------- |
| `parallel` | All tasks start simultaneously |
| `sequential` | Tasks run one at a time in order; first failure stops the job |
| `configurable` | Mode is read from the `JOB_EXECUTION_MODE` environment variable at runtime |

## Workflows

Workflows use DAG-based ordering to express dependencies between tasks. Tasks include `depends_on` (list of prerequisite task IDs) and `run_if` (conditional execution) fields.

**Create a workflow:**

```bash
curl -X POST "http://localhost:7070/workflows" \
-H "Content-Type: application/json" \
-d '{
  "tasks": [
    {"name": "fetch-data", "image_url": "docker.io/myorg/fetch:v1"},
    {
      "name": "process",
      "image_url": "docker.io/myorg/process:v1",
      "depends_on": ["<fetch-data-task-id>"],
      "run_if": "success"
    },
    {
      "name": "cleanup",
      "image_url": "docker.io/myorg/cleanup:v1",
      "depends_on": ["<fetch-data-task-id>"],
      "run_if": "failure"
    }
  ]
}'
```

**`run_if` values:**

| Value | Behavior |
| ----- | -------- |
| `success` _(default)_ | Task runs only if all dependencies completed successfully |
| `failure` | Task runs only if any dependency failed |
| `always` | Task always runs regardless of dependency outcome |

## Scheduling

The Manager includes a cron scheduler that checks for due tasks every 60 seconds. Tasks can be scheduled with a standard cron expression:

```bash
curl -X POST "http://localhost:7070/tasks" \
-H "Content-Type: application/json" \
-d '{
  "name": "hourly-report",
  "image_url": "docker.io/myorg/report:v1",
  "schedule": "0 * * * *",
  "timezone": "UTC",
  "is_recurring": true
}'
```

The Manager uses two scheduling mechanisms:

| Mechanism | Role | Behavior |
| --------- | ---- | -------- |
| **Round Robin** | Proplet selection | Selects which proplet receives the task, cycling evenly across alive proplets |
| **Priority ordering** | Task dispatch order | When multiple tasks are pending, higher-priority tasks (0–100, default 50) are dispatched first; ties broken by creation time |

Round Robin is the active proplet-selector. Priority ordering determines which pending task is dispatched next — it is not a separate proplet-selection algorithm.

Proplets are only eligible for scheduling if they are "alive" — i.e. a heartbeat was received within the liveliness threshold window.

## Task States

Tasks move through the following states during their lifecycle:

| Value | Name | Description |
| ----- | ---- | ----------- |
| `0` | `Pending` | Task created, not yet started |
| `1` | `Scheduled` | Task assigned to a proplet |
| `2` | `Running` | Task is executing on a proplet |
| `3` | `Completed` | Task finished successfully |
| `4` | `Failed` | Task encountered an error |
| `5` | `Skipped` | Task skipped (e.g. `run_if` condition not met) |
| `6` | `Interrupted` | Task was stopped externally |

## Proplet Management

The Manager tracks the health of proplets through MQTT heartbeats received on `m/{domain_id}/c/{channel_id}/control/proplet/alive`. A proplet is considered alive if its heartbeat was received within the last 10 seconds. The Manager stores the last 10 heartbeat timestamps per proplet.

## Architectural Components

### Middleware

The Manager wraps its service layer with three middleware layers:

- **Logging**: Logs every service method call with duration and errors.
- **Metrics**: Collects call count and latency via Prometheus.
- **Tracing**: Adds OpenTelemetry distributed tracing when `MANAGER_OTEL_URL` is configured.

### Storage

All task, proplet, job, and workflow state is persisted through a pluggable storage interface. Select the backend with `MANAGER_STORAGE_TYPE`:

| Backend | Type | Best for |
| ------- | ---- | -------- |
| `memory` _(default)_ | In-memory | Development and testing; state is lost on restart |
| `sqlite` | Embedded SQL | Single-node or edge deployments |
| `postgres` | Relational DB | Production deployments requiring durability |
| `badger` | Embedded key-value | High-performance edge deployments |

## Data Flow

### Task Execution

1. Client sends `POST /tasks` to create a task.
2. Client sends `POST /tasks/{id}/start` to start it.
3. Manager selects an alive proplet using the configured scheduler.
4. Manager publishes a start command to the proplet's MQTT topic.
5. Proplet fetches the Wasm binary (from Proxy or embedded in task), executes it, and publishes results.
6. Manager receives results, updates task state to `Completed` or `Failed`, and stores results.

### Proplet Registration

1. Proplet starts and publishes a discovery message to `control/proplet/create`.
2. Manager records the proplet ID and channel.
3. Proplet sends periodic heartbeats to `control/proplet/alive`.
4. Manager marks proplets as alive/dead based on heartbeat recency (10-second threshold).
