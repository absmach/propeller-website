---
title: Proxy Service
description: Service for fetching and distributing WebAssembly modules from OCI registries
---

The Proxy Service fetches WebAssembly modules from OCI-compliant registries, chunks them for efficient transfer, and publishes them over MQTT for Proplets to consume. It bridges HTTP (OCI registries) and MQTT (Propeller messaging) protocols.

## Why the Proxy

### Decoupling Binary Distribution

The Proxy separates binary fetching from task execution. Proplets request modules by name, and the Proxy handles registry authentication, download, chunking, and delivery. This keeps Proplets lightweight and focused on execution.

### Efficient Large File Transfer

MQTT has message size limits. The Proxy splits large WebAssembly modules into configurable chunks (default 512 KB) and streams them sequentially. Proplets reassemble chunks using the `ChunkPayload` structure:

```go
type ChunkPayload struct {
    AppName     string
    ChunkIdx    int
    TotalChunks int
    Data        []byte
}
```

### Registry Abstraction

The Proxy supports any OCI-compliant registry (Docker Hub, GitHub Container Registry, private registries) with optional authentication. Proplets don't need registry credentials—they just request modules by image URL.

## Running the Proxy

### With Docker Compose

The `docker-compose.yaml` includes the Proxy:

```yaml
proxy:
  image: ghcr.io/absmach/propeller-proxy:latest
  environment:
    PROXY_REGISTRY_URL: ${PROXY_REGISTRY_URL}
    PROXY_AUTHENTICATE: ${PROXY_AUTHENTICATE}
    PROXY_REGISTRY_USERNAME: ${PROXY_REGISTRY_USERNAME}
    PROXY_REGISTRY_PASSWORD: ${PROXY_REGISTRY_PASSWORD}
    PROXY_MQTT_ADDRESS: ${PROXY_MQTT_ADDRESS}
    PROXY_DOMAIN_ID: ${PROXY_DOMAIN_ID}
    PROXY_CHANNEL_ID: ${PROXY_CHANNEL_ID}
    PROXY_CLIENT_ID: ${PROXY_CLIENT_ID}
    PROXY_CLIENT_KEY: ${PROXY_CLIENT_KEY}
```

### Standalone

Build and run the binary:

```bash
make all && make install
propeller-proxy
```

## Architecture

The Proxy runs two concurrent streaming goroutines that communicate via an internal channel.

### HTTP Stream

The `StreamHTTP` method handles registry communication:

1. Receives container requests from the `containerChan` channel
2. Enforces concurrency limits (max 50 concurrent fetches)
3. Deduplicates in-flight requests to prevent redundant downloads
4. Fetches from the OCI registry using the ORAS library
5. Splits the binary into chunks and sends them to `dataChan`

The HTTP fetcher resolves the OCI manifest, finds the largest layer (the WASM binary), and downloads it. Authentication is configured once per Proxy instance.

### MQTT Stream

The `StreamMQTT` method handles Proplet delivery:

1. Receives chunks from `dataChan`
2. Publishes each chunk to the MQTT topic for Proplets
3. Tracks delivery progress per container
4. Logs completion when all chunks are sent

### MQTT Topics

| Topic Pattern | Direction | Purpose |
|--------------|-----------|---------|
| `m/{domain}/c/{channel}/registry/proplet` | Subscribe | Receives module requests from Proplets |
| `m/{domain}/c/{channel}/registry/server` | Publish | Sends chunks back to Proplets |

## Configuration

### Core Settings

| Variable | Description | Default |
|----------|-------------|---------|
| `PROXY_LOG_LEVEL` | Log level (`debug`, `info`, `warn`, `error`) | `info` |
| `PROXY_INSTANCE_ID` | Unique ID for this Proxy instance | — |

### MQTT Configuration

| Variable | Description | Default | Required |
|----------|-------------|---------|----------|
| `PROXY_MQTT_ADDRESS` | MQTT broker URL | `tcp://localhost:1883` | Yes |
| `PROXY_MQTT_TIMEOUT` | Operation timeout | `30s` | No |
| `PROXY_MQTT_QOS` | Quality of Service level | `2` | No |
| `PROXY_DOMAIN_ID` | SuperMQ domain ID | — | Yes |
| `PROXY_CHANNEL_ID` | Channel ID for MQTT | — | Yes |
| `PROXY_CLIENT_ID` | Unique client identifier | — | Yes |
| `PROXY_CLIENT_KEY` | MQTT authentication password | — | Yes |

### Registry Configuration

| Variable | Description | Default | Required |
|----------|-------------|---------|----------|
| `PROXY_REGISTRY_URL` | OCI registry URL | — | Yes |
| `PROXY_AUTHENTICATE` | Enable registry auth | `false` | No |
| `PROXY_REGISTRY_USERNAME` | Registry username | — | If auth enabled |
| `PROXY_REGISTRY_PASSWORD` | Registry password | — | If auth enabled |
| `PROXY_REGISTRY_TOKEN` | Access token (alternative to username/password) | — | If auth enabled |
| `PROXY_CHUNK_SIZE` | Chunk size in bytes | `512000` | No |

### Authentication

The Proxy supports two authentication methods:

**Token Authentication:**
```bash
export PROXY_AUTHENTICATE="true"
export PROXY_REGISTRY_TOKEN="your_token"
```

**Username/Password Authentication:**
```bash
export PROXY_AUTHENTICATE="true"
export PROXY_REGISTRY_USERNAME="your_username"
export PROXY_REGISTRY_PASSWORD="your_password"
```

## Using Hosted Modules

When using the Proxy, tasks specify an `image_url` instead of uploading binaries directly:

```json
{
  "name": "add",
  "inputs": [10, 20],
  "image_url": "docker.io/your-username/add.wasm"
}
```

The Manager forwards this to the Proplet, which requests the module from the Proxy. See the [Manager documentation](/docs/manager) for task creation details.

## Deploying a Local OCI Registry

For development, you can run a local registry:

```bash
docker run -d -p 5000:5000 --name registry registry:3.0.0
```

Push modules using `wasm-to-oci`:

```bash
wasm-to-oci push ./build/addition.wasm localhost:5000/your-namespace/addition.wasm
```

Configure the Proxy for the local registry:

```bash
export PROXY_REGISTRY_URL="localhost:5000"
export PROXY_AUTHENTICATE="false"
```
